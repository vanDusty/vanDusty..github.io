【面试】程序员的简历该怎么写？

发表于 2021-10-30 分类于 面试
本文字数： 2.6k 阅读时长 ≈ 5 分钟

简历是我们面试的敲门砖，简历基本属于面试流程的50%，好的简历面试官一看就能被吸引，大部分简历被扔到废纸篓。

程序员就应该拿技术说话，「技术总结」是一份程序员简历的重中之重。除了大神和大牛，普通的程序员如果能在叙述中体现自己的风格，在技术总结中展示鲜明的个人形象，会更吸引公司的注意，拿到更多的面试邀请。

一、小小的建议

推荐大家使用 Markdown 语法写简历，然后再将 Markdown 格式转换为PDF格式后进行简历投递。

  • 为什么用 Markdown 语法？
  1. 语法真的非常简单;
  2. 程序员的世界到处都是 Markdown，像简书，GitChat， GitHub，CSDN 等等都支持 Markdown 文档，正宗的官方技术文档都是使用 Markdown 来写的。

如果你对Markdown语法不太了解的话，可以花五分钟简单看一下 Markdown 语法

  • 为什么转换为PDF格式后进行投递

我们都知道，因为 Word 版本不同或者 Mac/Windows 系统等原因，如果不转成 PDF格式，你辛辛苦苦编辑的样式就可能到别人手上就变得惨不忍不。为了保证别人看到的样式跟你想要表达的一致，推荐转换为PDF格式后进行简历投递。

写简历的几个原则：

  1. 仔细核对，语句通畅，没错别字；
  2. 尽量避免主观表述，少一点语义模糊的形容词，除非是大公司大牛，已经有成果撑腰，否则慎用「熟悉… …」、「使用过… …」
  3. 简历上体现的项目细节，核心指标要有依据，能自圆其说，被认为在撒谎基本就GG；
  4. 不要过分夸大，包装自己的项目和技术，不懂的虚心请教，不要编或者瞎猜；
  5. 如果面试官专挑冷门，你简历完全没涉及的技术点问，可以喷他，这种公司不去也罢。

二、 两大法则

STAR 法则

  1. Situation： 事情是在什么情况下发生；
  2. Task:： 你是如何明确你的任务的；
  3. Action： 针对这样的情况分析，你采用了什么行动方式；
  4. Result： 结果怎样，在这样的情况下你学习到了什么。

简而言之，STAR 法则，就是一种讲述自己故事的方式，或者说，是一个清晰、条理的作文模板。不管是什么，合理熟练运用此法则，可以轻松的对面试官描述事物的逻辑方式，表现出自己分析阐述问题的清晰性、条理性和逻辑性。

FAB 法则

  1. Feature： 是什么；
  2. Advantage： 比别人好在哪些地方；
  3. Benefifit： 如果雇佣你，招聘方会得到什么好处。

简单来说，这个法则主要是让你的面试官知道你的优势、招了你之后对公司有什么帮助。

四、小结

公司和项目经验是否分开？

很多人喜欢公司和项目经历分开，我觉得其实没必要，本来工作经历就是跟公司绑定的。PS：仁者见仁智者见智，你有不同的想法，按照你觉得 OK 的改好了。

博客、Github 是否放到简历上？

简而言之：有亮点！如果你的博客或者 Github 上有原创比较有深度的文章或者给某些知名开源项目提交过有意义的代码，写上去会帮你加分。反之，则不建议放上去。

推荐一个在线编辑简历的网址 【冷熊简历】,他也给了一定建议，可以参考，按照相应的格式编写好你的简历，然后导出成 PDF 就可以了。
